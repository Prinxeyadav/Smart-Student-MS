🤖 AI Engine — Powered by FREA Pattern
4 intelligent algorithms run entirely in your browser. Every time Firestore fires an onSnapshot event, the AI automatically re-analyses all students. No API key. No server. This is the uniqueness of the FREA (Firebase Real-time Event-driven Architecture) pattern.

⚡ FREA Pattern — Firebase Real-time Event-driven Architecture (v4.0)

This system is built on the FREA Pattern — a unique architectural pattern developed for this project by Prince Kumar. Unlike traditional request-response systems, FREA uses Firestore's onSnapshot() listeners to push data changes to the UI instantly. Every write to Firestore automatically triggers the AI Engine re-analysis — no polling, no refresh, no server required. Data is isolated per user at users/{uid}/students enforced by Firestore Security Rules.

🤖 AI Engine — 4 Algorithms

1. Risk Predictor — Weighted scoring: 40% CGPA + 35% Attendance + 25% Weakest Subject → Risk Score 0–100
2. Anomaly Detection — Flags deviations from class averages; detects detained students & failing subjects automatically
3. Study Recommender — Generates personalised subject-level improvement tips based on individual scores
4. NLP Search — Plain-English queries like "show high risk students" parsed via keyword-rule engine

👨‍🎓 Student Portal — How it works

Login: Student enters Enrollment No. + password (set by faculty in the Add Student form)
Default password: If left blank, defaults to enrollment number in lowercase (e.g. 22it001)
What they see: Their own CGPA, SGPA, attendance ring, subject progress bars, semester summary, AI risk score & study recommendations
What they cannot do: Edit any data — fully read-only
Real-time: If faculty updates a grade, the student sees it instantly via FREA onSnapshot

🔥 Required Firestore Security Rules

Paste these rules in Firebase Console → Firestore → Rules → Publish:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Faculty: only access their own private student records
    match /users/{userId}/students/{studentId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }

    // Student login index: publicly readable (no login needed)
    // Only stores: name, stuPass, facultyUID, studentDocId — NO grades
    match /studentIndex/{enrollmentNo} {
      allow read: if true;
      allow write: if request.auth != null;
    }

  }
}

⚠️ After updating rules, faculty must re-save each student (Edit → Save) once so their record appears in the studentIndex. New students added after this fix work automatically.
